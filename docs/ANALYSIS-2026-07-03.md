# 全系统分析 — A股量化选股系统 (2026-07-03)

**范围**: 6107 行 Python, 40 文件, 2 数据库
**目的**: 识别根因, 阻断"改来改去"循环, 建立可审计的因子管理体系

---

## 一、架构现状

```
data/ (store.py 962行 God-class)
  market.db: stocks(5525), daily(16.7M), benchmark_daily(1573), lhb_detail(0), northbound_flow(0)
  trades.db: sim_trades(25), strategy_config(1)
factor/
  compute.py → FACTOR_REGISTRY(11) + FUNDAMENTAL_FACTOR_REGISTRY(5) = 16因子
  synth.py → equal_weight / ic_weighted
  stats_cache.py → factor_cache.json (扁平文件, 无版本)
pipeline.py (315行 run(), 7步串联)
risk/ → optimizer/ → execution/ → monitor/
web/ (Flask, 端口8521)
```

## 二、根因: 设计缺陷链

```
硬编码注册表 → 改因子只能改代码 → 改完没有结构化审计 →
下一个 session 不知道上次为什么删/加 → 重新加回来 →
因子数来回震荡(4→6→11→16→4→16) → IC低的因子稀释alpha →
回测亏损 → 怀疑因子不行 → 又去找新因子 → 循环
```

**核心矛盾**: 因子全生命周期（注册→评估→筛选→合成）没有一个环节入了数据库。
全部靠代码里的 dict + git commit message 维持，而 commit message 在多 session 协作中不可靠。

## 三、分层问题

### A. 数据设计层

| # | 问题 | 后果 |
|---|------|------|
| A1 | 因子元数据在 FACTOR_REGISTRY 硬编码 dict | 改因子=改代码, 无法审计 |
| A2 | 因子评估结果在 factor_cache.json 扁平文件 | 无历史版本, 无法追踪 IC 衰减时间线 |
| A3 | 相关性矩阵是单位矩阵(全零非对角) | alpha 合成假设 16 个因子彼此无关, 数学上荒谬 |
| A4 | stocks 表 20+ 列, 大部分 NULL | 列爆炸, 应拆 stock_financials |
| A5 | northbound_flow / lhb_detail 表全空 | 建了表但从未同步成功 |

### B. 业务逻辑层

| # | 问题 | 证据 |
|---|------|------|
| B1 | Alpha fallback 到 equal_weight | config 写 ic_weighted, 但 IC map 有 10 个因子为空时 fallback 到等权 |
| B2 | 16 个因子注册了, pipeline 实际只用 IC map 中有的 | 约 6-10 个参与合成 |
| B3 | 因子方向不确定 | compute_bp_ratio: "低BP=高分(A股成长溢价)" vs compute_momentum: "A股反转效应" 逻辑矛盾 |
| B4 | pipeline.py:142 top 30% 硬截断 | 5000×0.3=1500, 再加流动性过滤后仅剩几十只 |

### C. 代码质量层

| # | 问题 |
