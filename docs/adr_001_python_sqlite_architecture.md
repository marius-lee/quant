# ADR-001: Python + SQLite 架构决策

**日期**: 2026-05-31
**状态**: 已确认 (Accepted)
**决策者**: 系统架构总监 + 系统架构师
**类型**: 架构选型

---

## 背景

在完成全面代码审计后，针对是否将系统从 Python + SQLite 迁移到 C++ + MySQL 或 Go + MySQL 进行了评估。

## 决策

**保持 Python + SQLite 架构，不做迁移。**

## 理由

### 1. 数据规模匹配

| 指标 | 当前值 | SQLite 上限 |
|------|--------|-------------|
| 数据量 | 63万行 | 140TB |
| 数据库文件 | ~400MB | 140TB |
| 并发用户 | 1 (本地) | ~10 并发读 |

SQLite 仅用了理论上限的 0.0000002%，不存在容量瓶颈。

### 2. 工作负载特征

- **单用户**：没有并发竞争，MySQL 的事务隔离和连接池是浪费
- **研究密集型**：因子迭代速度 >> 运行速度
- **批量计算**：日级别分析，对延迟不敏感
- **读多写少**：每日只写入增量数据，查询远多于写入

### 3. 性能瓶颈分析

当前热点全部在 C/C++ 扩展中执行：

| 操作 | 耗时占比 | 实际执行层 |
|------|---------|------------|
| 数据拉取 | 70% | 网络 I/O |
| 因子计算 | 10% | NumPy (C 内核) |
| 模型训练 | 15% | LightGBM/XCBoost (C++ 内核) |
| Python 胶水 | <5% | Python |

换语言只能优化最不耗时的那部分。

### 4. 生态依赖性

量化系统的核心依赖：
- **LightGBM/XCBoost**: 仅 Python/R/C++ 有成熟绑定
- **pandas/NumPy**: Python 独占
- **akshare/tushare**: Python SDK
- **plotly/ECharts**: Web 可视化与 Python 无缝集成

Go 生态缺失以上全部，C++ 需要大量的 Python 绑定层。

### 5. 团队效率

```
因子研究迭代:  Python: 5分钟  vs  C++: 30分钟
ML模型实验:    Python: 直接调用  vs  Go: 无生态
代码量:        当前~5000行  vs  C++等价: ~15000行
```

量化研究最大的瓶颈是因子迭代速度，不是 CPU 周期。

### 6. 运维成本

```
MySQL = 安装 + 守护进程 + 账户管理 + 备份策略 + 网络配置 + 故障排查
SQLite = 一个文件，零运维
```

单用户本地系统引入 MySQL 是不必要的复杂化。

## 后果

### 正面

- 零运维开销，系统持续可用
- 因子研究迭代速度最大化
- 当前 400MB 数据量下，性能完全可接受
- 不需要学习新语言的量化生态系统

### 负面

- 不适用于多用户协作场景（当前无此需求）
- 如果数据增长到 100GB 以上，需要考虑升级

### 未来升级路径（当需要时）

```
1. SQLite → DuckDB (无代码修改，OLAP 快 10-100x)
2. pandas → Polars (API 相似，内存效率高 5-10x)
3. 单机瓶颈 → PostgreSQL (成熟生态)
4. CPU 瓶颈 → GPU 加速
5. 多人协作 → FastAPI + PostgreSQL + 定时调度
```

## 迁移触发条件

| 条件 | 当前 | 触发值 |
|------|------|--------|
| 数据量 | 400MB | >100GB |
| 并发用户 | 1 | >10 |
| 实时延迟需求 | 日级别 | <10ms |
| 写入频率 | 每天几百次 | 每秒万次 |
| 团队规模 | 单人 | >5人共享数据库 |

当前不满足任何一项触发条件。
