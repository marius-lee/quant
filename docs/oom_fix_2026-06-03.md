# OOM 修复 & 管线优化 — 2026-06-03

## 问题

auto_run 在 IC screening 步骤被 macOS OOM Killer SIGKILL。M1 8GB 可用 ~3GB，原管线峰值 5.4GB。

## 根因

| # | 问题 | 峰值 | 位置 |
|---|------|------|------|
| 1 | IC 全量加载 2847股×1084天×86因子 + rank拷贝 | 5.4GB | screener.py, screening.py |
| 2 | 3树模型 parallel=True 同时训练，各转 float64 | +1.8GB | trainer.py, ensemble.py |
| 3 | get_daily() 被调4次，3次冗余 | +900MB 浪费 | loader/screener/ranker |
| 4 | get_daily 无日期边界 | +200MB 浪费 | store.py |
| 5 | OOM 后 status 永远 "running" | — | auto_run.py |
| 6 | launchd plist 缺 TUSHARE_TOKEN | — | plist |

## 修复

### 1. 批量日期流式 IC 筛选（-5.3GB）

沿日期维分块，每批 50 天 × 全量 2847 股。批内全局 rank per date → IC 精确等价全量。

- 新函数: `factor/screening.py::screen_factors_streaming()`
- `engine/screener.py` 全局路径改为调用新函数
- 内存: 5.4GB → ~100MB

### 2. 训练串行化（-1.2GB）

- `engine/trainer.py`: 删除 `parallel=True`
- `strategy/ensemble.py`: 串行 + 模型间 `gc.collect()`

### 3. 共享 OHLCV 缓存（-900MB）

- `engine/loader.py`: `get_training_data()` 返回 `raw_daily`
- `screener`/`ranker`/`factor_strategy` 接收复用，不再各自查询

### 4. float32 全程 + 日期绑定

- `data/repository.py`: `load_batch` 返回 float32
- `data/store.py`: `get_daily` 返回 float32
- 单日查询跳过 MAX_PARAMS 分块

### 5. 崩溃安全状态

- `auto_run.py`: 启动时检测 stuck "running" → 自动重置 "failed"

### 6. launchd token

- `com.quant.analysis.plist`: 添加 TUSHARE_TOKEN 环境变量

### 7. 实时训练（train_split 调整）

- `config.yaml`: `train_split` 从 0.7 → 0.92
- 训练窗口从 2023-06~2024-06 变为 2025-09~2026-01（滞后从 2 年缩到 5 个月）

### 8. 进度条

- IC 筛选、因子衰减、模型训练各阶段终端进度条

## 配置决策

| 参数 | 旧值 | 新值 | 原因 |
|------|------|------|------|
| train_split | 0.7 | 0.92 | 训练更接近当下，减少模型滞后 |
| train_window | — | 252 | 保持，约1年交易数据足够 |
| slippage | 0.003 | 0.01 | 小盘/妖股实盘滑点更真实 |
| tushare_token | config.yaml | ${ENV_VAR} | 安全，不提交到 git |
| ml_weight | 0.2 | planner 动态 | 按资金阶段调整 |
| demon 权重 | 80% 固定 | 条件加成(threshold=0.3) | 妖股失效时 ML 主导 |
| max_drawdown | — | planner 动态 | 按阶段 (0.15~0.80) |
