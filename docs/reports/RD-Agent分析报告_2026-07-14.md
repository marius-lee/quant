 ## 一、RD-Agent 项目概况
 
 RD-Agent 是微软研究院开源的自动化 R&D agent 框架，核心思想是将"研究与开发"拆分为两个可自动迭代的过程：
 
 - **R (Research/Proposal)**: 基于历史反馈提出新假设（Hypothesis），并生成 Experiment
 - **D (Development)**: 将 Experiment 转化成可执行代码，在 Docker 沙箱中运行，收集反馈
 
 该框架在 MLE-bench（75 个 Kaggle 竞赛）中排名第一，其量化场景（RD-Agent(Q)）被 NeurIPS 2025 接收，展示了因子-模型联合优化的范式。
 
 ### 代码规模
 
 | 维度 | RD-Agent | 我们的项目 |
 |------|----------|-----------|
 | Python 文件数 | ~529 个（仅 rdagent/） | ~40 个（仅 quant/，不含 .venv） |
 | 核心包 | `rdagent/core/` (13 个抽象类) | 分层架构 7 layers + evaluation/ + backtest/ |
 | 文档体系 | Sphinx + RST + ReadTheDocs + 29 个子 README | 散落的 Markdown + 31 个 ADR |
 | CI/CD | GitHub Actions, mypy, ruff, pytest, coverage | pytest 67 tests, 无 CI |
 | 配置管理 | pydantic-settings 类型化配置 | YAML + `_require_cfg()` |
 | 包管理 | pyproject.toml + pip installable | requirements.txt |

 ## 二、架构对比与借鉴价值
 
 ### 2.1 RD-Agent 核心抽象层
 
 RD-Agent 的核心抽象定义在 `rdagent/core/`：
 
 ```
 Scenario (场景描述) → Hypothesis (假设) → Experiment (实验)
     ↑                  ↑                    ↑
   背景/数据/接口       LLM 生成             任务序列 + 工作空间
 
 Developer (开发者)    ←  实现 Experiment
 EvoAgent (进化代理)   ←  多轮迭代 Developer
 Evaluator (评估器)    ←  收集反馈 (Feedback)
 Trace (追踪图)       ←  DAG 记录历史实验
 ```
 
 核心循环：
 ```
 HypothesisGen.gen(trace) → Hypothesis
 Hypothesis2Experiment.convert(hypothesis, trace) → Experiment
 Developer.develop(exp) → executed Experiment
 Experiment2Feedback.generate_feedback(exp, trace) → Feedback
 Trace.sync_dag_parent_and_hist(exp, fb)
 ```
 
 ### 2.2 与我们项目的对比
 
 | 概念 | RD-Agent | 我们的项目 |
 |------|----------|-----------|
 | 实验记录 | `Trace` (DAG 图, 记录每次实验的假设-实现-反馈) | 无正式实验追踪；`evaluation_runs` 表只记录相位结果 |
 | 假设验证 | `Hypothesis` 类承载假设理由和检验 | 无假设概念；因子通过 "IC > 阈值" 来判断好坏 |
 | 工作空间隔离 | `FBWorkspace` — 每个实验独立目录，可 checkpoint | `pipeline.py` 所有步骤共享环境 |
 | 多 round 迭代 | `EvoAgent.multistep_evolve()` — 最多 max_loop 轮 | 单次评估；无闭环反馈 |
 | 知识积累 | `KnowledgeBase` / `RAGStrategy` — 从历史 trace 学习 | 无知识库；无跨 run 学习 |
 | 选择策略 | `CheckpointSelector` / `SOTAexpSelector` — 从 trace 选节点 | 无 |
 
 ### 2.3 量化场景 (qlib/) 的架构
 
 RD-Agent 量化场景分两套独立模块：
 
 **因子模块 (factor)**:
 - `QlibFactorExperiment` — 继承通用的 `FactorExperiment`，注入 Qlib 模板
 - `QlibFactorScenario` — 场景背景、数据描述、输出格式
 - `FactorHypothesisGen` — 基于 trace 生成因子假设
 - `FactorCoder` — 在 Docker 中实现因子代码
 - `FactorRunner` — 在 Docker 中跑 Qlib 回测
 
 **模型模块 (model)**: 类似结构，针对 Qlib 模型
 
 **联合优化**:
 - `QlibQuantHypothesisGen` — 用 Bandit 或 LLM 决策本轮优化因子还是模型
 - `QlibQuantScenario` — 统一的因子+模型场景描述
 - `EnvController` — 基于指标的自适应权重
 
 ### 2.4 值得借鉴的架构设计
 
 #### 2.4.1 Experiment + Trace 体系（高优先级）
 
 **现状**: 我们的评估管线（evaluation/ 五阶段）产出阶段结果存入 `evaluation_runs` 表，但无"实验"的正式概念。回测（backtest/loop.py）产出 Sharpe/MDD，但和评估管线之间是割裂的——回测结果不输入评估管线做闭环，评估管线的输出也不影响回测参数。
 
 **RD-Agent 的做法**: 每次 idea 生成 → 实现 → 执行的完整生命周期称为一个 `Experiment`，存入 `Trace`。Trace 是个 DAG（用 `dag_parent` 记录父子关系），支持：
 - 回溯到任意历史节点从那里继续改进
 - 按 decision=True/False 找 SOTA
 - 多分支并行
 
 **建议**: 引入 "实验" 概念作为统一的数据结构，替换当前散落的状态：
 - `evaluation_runs` 表加 `experiment_id`, `hypothesis`, `parent_id` 字段
 - 每次 factor_registry 状态变更（using→monitoring→retired）作为一个 `Experiment` 记录
 - 回测的每个变体（参数、因子组合）也作为 Experiment
 
 #### 2.4.2 Hypothesis-Driven Development（中优先级）
 
 **现状**: 我们的因子被分为 active/deprecated/monitoring/retired，但没有记录"为什么这个因子被认为有效"或"为什么它失效了"。
 
 **RD-Agent 的做法**: `Hypothesis` 类包含 `hypothesis`, `reason`, `concise_observation`, `concise_justification`, `concise_knowledge`。
 
 **建议**: 
 - 因子 registry 加 `hypothesis` 和 `reason` 字段
 - 每个因子入库时记录"为什么认为它有效"（从研报/历史 IC/理论推导）
 - 因子退场时记录具体原因（IC 衰减到什么程度、哪个市场环境不再适用）
 
 #### 2.4.3 Workspace 隔离（低优先级）
 
 **现状**: 所有评估和回测共享同一个 market.db，无隔离。
 
 **RD-Agent 的做法**: 每个 Experiment 在独立目录（`FBWorkspace`）下运行，支持 checkpoint/recover。通过 Docker 隔离运行环境。
 
 **建议**: 短期内不需要 Docker 隔离（我们不做代码生成），但可以考虑给回测变体独立的 workspace（数据快照 + 参数配置归档）。
 
 #### 2.4.4 演化策略 (EvolvingStrategy)（中优先级）
 
 **现状**: 因子评估是一次性的（跑五阶段 → 通过/不通过）。没有多轮反馈闭环。
 
 **RD-Agent 的做法**: `EvoAgent` 最多 max_loop 轮迭代，每轮：RAG 查知识 → evolve（生成/改进）→ evaluate（收集反馈）→ 更新 trace → 知识自生成。
 
 **建议**:
 - 将因子参数调整作为一个自动迭代过程（一个因子参数化后，在网格上搜索最优窗口，类似 Optuna）
 - 每次迭代产生反馈（IC 变化、衰退速度），存入 trace
 - 这一点与 ADR 029 的 P2（自动参数搜索 Optuna）直接对齐
 
 #### 2.4.5 知识库 (KnowledgeBase / RAG)（高优先级）
 
 **现状**: 我们有大量因子研报（docs/research/ 下 20+ 篇），但没有结构化的知识提取和检索。
 
 **RD-Agent 的做法**:
 - `KnowledgeBase` 是持久化知识存储
 - `RAGStrategy` 在执行前查询相关知识
 - `knowledge_self_gen` 执行后从 trace 自动提取新知识
 
 **建议**:
 - 构建"因子卡片"（factor card）知识库：每个因子一个 JSON，记录公式、IC 统计、数据依赖、评估历史、适用/不适用市场环境
 - 在开发新因子时自动检索相似因子（避免重复）
 - 因子失效时自动记录"市场环境 → 因子表现"映射

 ## 三、RD-Agent 量化场景 (qlib/) 的算法
 
 ### 3.1 Bandit 动作选择
 
 在因子-模型联合优化中，`QlibQuantHypothesisGen` 使用 `EnvController` 决定每轮优化因子还是模型：
 
 ```python
 if action_selection == "bandit":
     metric = extract_metrics_from_experiment(trace.hist[-1][0])
     prev_action = trace.hist[-1][0].hypothesis.action
     trace.controller.record(metric, prev_action)
     action = trace.controller.decide(metric)
 elif action_selection == "llm":
     # LLM 根据历史反馈决策
     ...
 elif action_selection == "random":
     action = random.choice(["factor", "model"])
 ```
 
 **对我们项目的意义**: 这不是直接可用的（我们不做模型优化，只做因子），但选择逻辑本身可以借鉴——例如，用 multi-armed bandit 决定因子评估的资源分配（哪个新因子值得更多算力验证）。
 
 ### 3.2 因子提取 from 研报
 
 RD-Agent 支持从 PDF 自动提取因子公式并转化成代码：
 - `factor_experiment_loader/pdf_loader.py` + `json_loader.py`
 - LLM 读研报 → 提取公式 → 生成 Qlib 表达式 → Docker 中验证
 
 **对我们项目的意义**: 我们从 12 家券商研报中挖掘了 N 个因子，目前都是手动完成的。如果有结构化因子卡片库 + LLM 辅助提取，这个流程可以加速。
 
 ### 3.3 Factor Benchmark 模块
 
 `rdagent/app/benchmark/factor/eval.py` — 独立的因子评估基准：
 
 ```python
 test_cases = FactorTestCaseLoaderFromJsonFile().load(bench_data_path)
 eval_method = FactorImplementEval(
     method=generate_method,
     test_cases=test_cases,
     catch_eval_except=True,
     test_round=bench_test_round,
 )
 res = eval_method.eval(eval_method.develop())
 ```
 
 **设计特点**:
 - 测试用例和评估逻辑解耦
 - `FactorImplementEval` 是一种 `Evaluator` 实现
 - 结果直接输出为结构化 report
 
 **对我们项目的意义**: 我们的 evaluation/ 五阶段比这个更完整（有 CPCV + PBO + walk-forward），但缺少"基准测试"的概念——即把多个因子生成方法在同一套测试数据集上公平比较。

 ## 四、可以改进我们项目的具体算法和组件
 
 ### 4.1 因子卡片 (Factor Index Card) — 高价值、可落地
 
 **来源**: RD-Agent 的 `factor_experiment_loader/json_loader.py` + 多轮迭代反馈
 
 **设计**:
 ```json
 {
   "factor_id": "momentum_20d",
   "name": "20日动量",
   "category": "momentum",
   "formula": "close / close.shift(20) - 1",
   "data_deps": ["daily.close"],
   "ic_mean_12m": 0.034,
   "ic_std_12m": 0.12,
   "icir_12m": 0.28,
   "half_life_days": 15,
   "decay_trend": "declining",
   "status": "monitoring",
   "status_history": [
     {"from": "active", "to": "monitoring", "date": "2026-06-01", "reason": "IC < 0.02 for 3 consecutive weeks"}
   ],
   "market_regime_performance": {
     "bull": {"ic_mean": 0.045},
     "bear": {"ic_mean": 0.012},
     "volatile": {"ic_mean": 0.022}
   },
   "correlations": {"momentum_60d": 0.72, "rsi_14d": -0.15},
   "last_evaluated": "2026-07-10"
 }
 ```
 
 **与现有系统的整合**:
 - `factor/registry.py` 已管理因子状态；卡片是其结构化扩展
 - `factor/ic.py` 的 `ic_history` 已有各因子 IC 时间序列
 - 卡片在 `factor/ic.py` 中自动生成/更新，替代当前分散在各处的状态判断
 
 ### 4.2 闭环反馈与自动参数搜索 — ADR 029 P1/P2 的推进路径
 
 **现状**: ADR 029 规划了四层回测架构，P1 (walk-forward) 和 P2 (Optuna) 待实现。
 
 **RD-Agent 的 EvoAgent 模式给出的路径**:
 
 ```
 因子初值 (当前 evaluation 输出)
   → RAG 检索相似因子 + 历史知识
   → 参数扰动 (窗口、因子组合权重)
   → 回测验证 → 反馈 (Sharpe 变化、IC 变化)
   → 更新 Trace
   → 下一轮...
 ```
 
 这个循环中的"回测验证"就是 ADR 029 要做的 walk-forward。区别是：RD-Agent 用 `Experiment` 结构化了整个过程，我们可以用 `Experiment` 概念统一 evaluation/ 和 backtest/ 两个模块。
 
 ### 4.3 知识库驱动的因子发现 — 中价值、依赖 LLM
 
 **建议**: 将 docs/research/ 下的分散研报整合为结构化的因子知识库：
 
 1. 每个研报因子抽取为给定格式的 JSON
 2. 构建向量索引（用 embedding 做相似因子检索）
 3. 在新因子开发时，检索是否有重复/近似因子
 4. 在有新研报时，半自动提取候选因子
 
 这个需要 LLM API 成本（embedding + chat），适合作为可选增强，而非核心依赖。

 ## 六、不需要搬过来的东西
 
 | RD-Agent 特性 | 不适合的原因 |
 |-------------|------------|
 | Docker 沙箱代码生成 | 我们不生成代码，所有因子是预先实现的 |
 | Qlib 依赖 | 我们有自建的 market.db + compute.py 计算管线 |

 | Streamlit UI | 我们已有 Flask + 前端，不需要替换 |
 | LiteLLM 多模型后端 | 暂不需要 LLM 集成 |
 | MLE-bench 评估 | 我们不做 ML 竞赛 |

 ## 七、五条核心建议（按优先级）
 
 ### P0: 因子卡片知识库
 
 替代当前分散在各处（registry, ic.py, ad hoc 判断）的因子元数据。因子卡片是一个 JSON schema，记录每个因子的公式、IC、数据依赖、适用/不适用市场环境、状态变更历史。
 
 **落地方案**: 
 - 在 `factor/` 下新建 `cards/` 目录
 - 每个活跃因子一个 JSON 文件
 - `factor/ic.py` 在每个评估周期完成后自动更新卡片
 - 卡片格式参考上文 4.1 节
 
 ### P1: 引入 Experiment + Trace 概念
 
 统一 evaluation/ 和 backtest/ 两个模块，让每次评估/回测都作为一次"实验"被追踪，形成可回溯的 DAG。
 
 **落地方案**:
 - `evaluation_runs` 表升级为 `experiments` 表（加 hypothesis, parent_id, status 字段）
 - 回测层的每次变体也写入同一张表
 - 新增 `core/trace.py`：轻量 Trace 类（比 RD-Agent 的更简单，只保留 DAG 父关系和 SOTA 查询）
 
 ### P2: 标准化文档结构
 
 按上文 5.3 节 Phase 1-3 执行。从根目录 MARKDOWN 的清理开始，同步推进。
 
 **最低方案** (今天就做): 新建 `CONTRIBUTING.md` 和 `factor/README.md`, `backtest/README.md`, `evaluation/README.md`。
 
 ### P3: Walk-forward 闭环 (ADR 029 P1/P2)
 
 将 evaluation/ 的五阶段输出和 backtest/ 的 loop 连接起来，形成闭环：
 
 ```
 evaluation/ (五阶段) → 因子通过认证 → factor_registry 更新
 backtest/ (四层)     → 回测结果 → 反馈到 evaluation/ (重新校准阈值)
 ```
 
 **落地方案**:
 - `evaluation/phase6_backtest.py` 已经存在但未落地；补全其实现
 - 回测输出写入 experiments 表（P1 后可用），由 evaluation/ 查询
 - walk-forward 的回测结果自动更新因子卡片的 IC 统计
 
 ### P4: 建立 CI/CD 和质量工具链
 
 从 RD-Agent 的 `pyproject.toml` 直接参考：
 
 ```
 - mypy (check_untyped_defs, disallow_untyped_defs)
 - ruff (line-length=120, select=["ALL"])
 - pytest (durations, markers)
 - coverage (fail_under=80)
 - pre-commit (husky + lint-staged)
 ```

 ## 八、讨论点（需要你来决策）
 
 1. **因子卡片 vs. factor_registry**: 卡片是完全替代 registry 还是共存？卡片是 JSON 文件还是数据库表？
 
 2. **Experiment 追溯粒度**: 每次 IC 评估算一次实验？还是只有状态变更（active→monitoring）才算？还是每次 pipeline 跑完算一次？
 
 3. **文档发布**: 要不要搭 Sphinx + ReadTheDocs？还是先用 Markdown + GitHub Wiki 过渡？
 
 4. **LLM 引入时机**: 因子卡片结构化后发现哪些应用场景可以受益于 LLM？（因子去重？因子发现？因子归因？）
 
 5. **pyproject.toml**: 要不要立刻把 requirements.txt 迁移到 pyproject.toml？这个改动会影响 .venv 的依赖管理。
