# 交易执行时机分析（修订版）

## 一、现状问题

```
08:30  signals    生成目标持仓 → 写 daily_signals
09:30  execute    一次性读取 → 一次性市价买入 → 结束
                  _timed_loop 触发放行 run_fn 一次就退出
09:35  monitor    持续到 14:55（只做止损卖出）
```

两个问题交织在一起：
1. **执行次数**: 每天只买一次，而不是持续寻求更好价格
2. **执行价格**: 只用快照市价，没有限价逻辑

第一份报告把这两个问题混在一起谈了。分开来看。

## 二、业界在 10:00-14:30 之间做什么

### 2.1 结论：大部分机构全天都在交易

| 机构类型 | 10:00-14:30 是否有买盘 | 做法 |
|---------|:---:|------|
| 公募量化 | ✅ 有 | T 日收盘出信号，T+1 日开盘到收盘之间用 VWAP/TWAP 执行，贯穿全天 |
| 头部私募（幻方/九坤） | ✅ 有 | 日内多次信号更新，10:00/11:30/14:00 各有一次 alpha 重算，有信号就下单 |
| 券商自营 | ✅ 有 | 同私募，且通常有多个独立策略并行，各自下单互不干扰 |
| 外资（Two Sigma/WQ） | ✅ 有 | 全天候算法执行，Almgren-Chriss 框架下每 5-15 分钟拆单一次 |
| 散户/小私募 | ✅ 有 | 手动或半自动，看到价格合适就买 |

**没有哪家机构的量化系统是"9:30 买完就关机的"。**

### 2.2 他们具体怎么做

以华泰金工 (2021) 对 A 股日内执行的研究为基础：

**（1）VWAP 基准执行（公募常用）**

```
08:45  拿到目标持仓清单
09:30  启动 VWAP 算法
09:30-14:55  算法按日内成交量预测曲线拆分订单
             每 5-15 分钟下一部分单
             成交量大的时段（开盘、尾盘）占比高
             成交量小的时段（午盘前后）占比低
14:55  检查剩余未成交 → 市价补完
```

关键点：**全天每个时段都在下单**，只是下单量按预测的成交量曲线分配。10:00-14:30 中间成交量相对低，但仍然有成交发生。

**（2）Implementation Shortfall（头部私募常用）**

```
08:45  拿到目标持仓
09:30  设定 arrival price（开盘价或信号生成时的参考价）
09:30-14:55  每 N 分钟评估：
             - 已成交部分 vs 未成交部分
             - 市场价格 vs arrival price
             - 动态调整限价/市价策略
             - 平衡"等待更好价格"和"价格拉远错过仓位"的 tradeoff
14:55  收盘前清空所有未成交订单
```

这是 Almgren-Chriss (2001) 框架在实盘中的应用。核心思想：**你等得越久，可能拿到更好的价格，但也可能价格走远了更亏**。最优解是动态调整的，不是固定时间点。

**（3）限价单 + 被动成交（灵均/部分私募小策略）**

```
09:30  挂限价买单（价格略低于当前价，如 -0.3% ~ -0.5%）
09:30-14:50  限价单挂在交易所等待成交
             价格波动触及限价 → 自动成交（这是在 10:00-14:30 之间发生的）
             每小时检查一次未成交订单：
               - 如果市场向有利方向移动 → 调整限价更激进一些
               - 如果市场向不利方向移动 → 考虑取消重挂或转市价
14:50  仍未成交的 → 取消 + 市价补单
```

关键：**限价单在 09:30 挂出后，在 10:15、11:20、13:45 等任何时刻都可能被触发成交**。10:00-14:30 不是"不操作"，而是"限价单在交易所被动等待，同时定期调整未成交部分"。

## 三、当前代码 vs 业界标准

| 维度 | 当前代码 | 业界标准 | 差距 |
|------|---------|---------|:---:|
| 下单次数 | 1 次 (09:30) | 持续（按 VWAP 曲线或限价等待） | 大 |
| 下单方式 | 市价（快照价） | 限价为主，市价为辅 | 大 |
| 价格优化 | 无（接受任何价格） | 限价等待 + VWAP 基准 | 大 |
| 订单管理 | 无（一次发射） | 监控、调整、补单 | 大 |
| 未成交处理 | 不存在（全部当时成交） | 尾盘补单机制 | 大 |

## 四、建议方案

针对 ¥5,000 规模、1-3 只股票的场景，VWAP 和 Implementation Shortfall 都太重（拆单没意义）。**适合的是简化版限价窗口 + 被动成交 + 尾盘补单**。

```
08:30  signals    生成目标持仓（不变）

09:30  place      为每只目标股挂限价买单
                  限价 = min(当前买一价, 昨日收盘 × 0.998)
                  含义：不高于昨收 0.2% 折扣，且不高于当前买入价
                  写订单到内存/轻量表中

09:30-14:50        限价单在交易所等待
                    ← 市场价触及限价 → 自动成交 ← 这发生在 10:15, 13:30 等任意时刻
                    每 30 分钟: 检查未成交订单
                    如果价格已离开限价 > 1% → 取消重挂（调整限价）
14:50  fill        未成交的 → 取消限价 → 市价买入（确保收盘前有仓位）
```

**这个方案和你第一反应"为什么不在便宜时买入"完全一致**：
- 如果 10:30 价格跌了 → 限价单可能在这时成交（买得更便宜）
- 如果全天都没跌到限价 → 14:50 市价补（至少买进去了）
- 09:30 市价买入的弊端被消除：不再强迫在开盘波动最大时出手

**这个方案里，10:00-14:30 之间：**
- 限价单持续挂在交易所，随时可能成交 ← 这是买盘操作
- 每 30 分钟检查未成交订单 ← 这是订单管理
- 不是"不操作"，而是"不主动进攻，等待价格来找你"

## 五、讨论点

1. **限价公式**: `min(买一价, 昨收×0.998)` — 这个 0.2% 折扣是否合理？A 股日波动中位数 ~2%，0.2% 意味着中等概率成交。如果太激进（比如 0.5%），可能全天不成交全靠尾盘补。

2. **调整频率**: 每 30 分钟检查一次是否合适？太频繁会增加 API 调用成本（新浪限频），太少会错过调整窗口。

3. **卖出逻辑**: 当前 monitor 是持续监控的（5s 一次拉行情），买和卖不对称。如果用限价窗口做买入，卖出是否也该做类似改造？还是保持卖出优先（快卖慢买是合理的风控原则）？

4. **Monitor 和 Execute 的关系**: 当前 monitor 只卖不买。如果 execute 改为"挂限价 + 持续管理"，execute 和 monitor 的边界就模糊了。可能需要合并或重新分工。

5. **实现复杂度**: 当前 execute 是一次性调 `_timed_loop` 触发 `_run` 一次。改为持续订单管理需要重构调度框架。是否值得先做最小可行版（比如：09:30 挂限价，14:50 补单，中间不调整）再迭代？
