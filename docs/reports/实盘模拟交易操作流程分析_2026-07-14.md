 # 实盘模拟交易操作流程分析 — 业界标准对比与改进建议

 > 2026-07-14 | 全量代码审计 + 文档综合分析

 ---

 ## 一、当前业务流程全景

 ### 1.1 日频调度五任务序列

 ```
 08:30  signals      信号生成（盘前）
   │      └─ generate_signals(status_filter="using")  ← using = active + monitoring
   │      └─ 因子计算 → Alpha → Risk → Optimizer → 目标持仓
   │      └─ 写入 daily_signals 表（持久化，重启安全）
   │
 09:30  execute      交易执行（开盘）
   │      └─ 从 daily_signals 表读取今日目标持仓
   │      └─ delta 调仓（目标 vs 当前）= compute_trades()
   │      └─ T+1 检查、除权检测、止损检查
   │      └─ 通过 ExecutionEngine.execute() 写入 trades.db
   │
 09:35-14:55  monitor  盘中风控守护（持续循环）
   │      └─ 每 30s 检查：总资产回撤 / 熔断 / 止损止盈
   │      └─ 每 5s 拉一次新浪实时行情（限频）
   │      └─ 触发止盈/止损 → 直接执行卖出
   │
 15:30  attribution  盘后归因
   │      └─ Brinson 归因（配置/选股/交互效应）
   │      └─ IC 衰减检测：5日滚动窗口，衰减 >30% → active→monitoring
   │      └─ monitoring 持续衰减 >10 天 → monitoring→retired
   │      └─ 基准对比（000300 沪深 300）
   │
 周六 06:00  weekly_eval  周度因子评估
          └─ 强制刷新 IC 缓存（Grinold & Kahn 标准：周度 IC 更新）
          └─ 因子权重重新计算
 ```

 ### 1.2 因子状态机全生命周期

 ```
 registered ──→ candidate ──→ retired ──→ 正式评估 Phase 2/3/4 ──→ active
                                                                         │
                                                        ┌────────────────┘
                                                        │ IC 衰减 >30%
                                                        ▼
                                                  monitoring ──→ retired
                                                  (观察期)      (持续 IC 衰减 >10d)

 快捷别名:
   using      = active + monitoring       (参与盘前信号生成)
   backtesting = registered + candidate + retired   (参与诊断/评估)
   rejected   = 永久剔除，永不参与任何流程
 ```

 关键代码位置：
- `factor/compute/_registry.py:16-18` — `_resolve_statuses()` 决定 `using` 和 `backtesting` 的实际含义
- `quant/scheduler/attribution.py:71-127` — 每日 15:30 自动检测 IC 衰减并执行降级
- `data/repos/factor_repo.py:16` — `VALID_STATUSES` 定义 8 个合法状态

 ### 1.3 两阶段评估管线（离线，非日频）

 两步架构（HANDOFF #2026-07-12#1）：

 1. **回测诊断** (`backtest/diagnostics.py`) — 因子"快照"，从 backtesting 池诊断 keep/boost/drop
 2. **正式评估** (`evaluation/` 五阶段) — 因子"认证"：
    - Phase 1: 数据准备
    - Phase 2: 单因子检验（|IC|≥0.02, ICIR≥0.25, t≥2.0, 半衰期≥20d）
    - Phase 3: CPCV Walk-Forward + PBO（PBO<0.3 + 夏普衰减<50%）
    - Phase 4: 扣费后回测验证
    - Phase 5: 持续监控 + `sync_factor_status()` 同步 rejected/active

 ## 二、业界标准对照

 ### 2.1 与业界共识的符合度

 | 维度 | 业界标准 | 当前项目 | 状态 |

| **数据划分** | 训练/验证/测试三段分离（60/20/20 或 walk-forward 滚动） | 离线评估已实现 walk-forward CPCV + PBO（Phase 3） | ✅ 符合 |
| **IC 门槛** | \|RankIC\|≥0.02（券商共识底线），≥0.05（优秀） | \|IC\|≥0.02（Phase 2），monitoring≥0.015 | ✅ 符合 |

| **幸存者偏差** | 包含退市股票 | data/store.py sync_delisted_stocks() | ✅ 符合 |

 ## 三、逐模块深度分析

 ### 3.1 信号生成 (08:30)

 **当前实现**: `quant/scheduler/signals.py` → `pipeline.generate_signals(status_filter="using")`

 **分析**:
- 因子计算：`compute_all_factors` 遍历所有 `using` 状态的因子，逐个计算
- Alpha：`AlphaModel.combine()` + `rank()` 用 IC 加权合成
- 风险：`neutralize()` 行业/市值中性化 + `covariance_matrix()` Ledoit-Wolf 协方差
- 优化：`PortfolioConstructor.construct()` 头寸分配

 **业界标准流程**（Grinold & Kahn 1999；华泰/中信建投金工研报）：
 1. 因子截面标准化（z-score）
 2. 因子合成（IC 加权 / 等权 / ML）
 3. 风险模型（Barra 多因子风险模型 / 统计协方差）
 4. 组合优化（均值-方差 / Kelly / 风险平价）
 5. 约束：行业暴露、市值暴露、换手率上限

 **我们的流程基本符合**，差异在于：
- 我们 IC 更新是 **周度**（周六 06:00），业界通常是**周度或月度**。周度合理。
- Alpha 模型对齐 Grinold & Kahn，IC 加权合成标准做法。

 ### 3.2 交易执行 (09:30)

 **当前实现**: `quant/scheduler/execute.py` → `pipeline.execute_signals()`

 **分析**:
- 从 `daily_signals` 表读取信号（持久化）
- 如无信号 → 直接拒绝执行，**不作 fallback** ← 这是正确的
- 获取 Sina 实时行情（开盘价），报价为空 → 拒绝执行
- 止损检查在调仓前完成（正确顺序）
- 计算目标 vs 当前持仓的 delta，生成 buy/sell 订单
- 通过 `ExecutionEngine.execute()` 写入 `trades.db`

 **与业界对比**:
- 止损在调仓前执行 ✅ — 避免止损资金用于新买入（金融意义上正确：止损是风控，优先级高于新开仓）
- 无信号拒绝执行 + 报价为空拒绝 ✅ — 零 fallback 原则
- 但调仓算法 `compute_trades()` 使用贪心排序，未考虑组合层面约束同时优化 — ⚠️ 这在学术界（Markowitz 1952）和业界（Barra Optimizer）都是次优的

 ### 3.3 盘中风控 (09:35-14:55)

 **当前实现**: `quant/scheduler/monitor.py`

 **分析**:
- 总资产回撤 >5% 告警，>5% 熔断
- 止损/止盈：通过 `execution/stop_loss.py` 的 `RiskManager.check()` 判断
- 每 5s 拉一次行情（限频保护）
- 触发后直接执行卖出

 **与业界对比**:
- 熔断机制 ✅ — 业界普遍做法（机构常用 5-10% 单日熔断）
- 止盈止损 ✅ — 标准配置
- 但缺少以下业界常见监控项：
  - **单行业/单票集中度监控** ⚠️
  - **VaR 实时跟踪** ⚠️（我们有 `risk/var.py`，但盘中监控未集成）
  - **交易频率监控**（防止过度交易） ⚠️
  - **流动性监控**（避免买卖流动性差的股票） ⚠️

 ### 3.4 盘后归因 (15:30)

 **当前实现**: `quant/scheduler/attribution.py`

 **分析（三部分）**:

 #### A. Brinson 归因
- 按行业分组计算行业收益，计算配置/选股/交互效应
- **问题 1**: 基准权重是等权（`1/len(all_secs)`），不是市值加权——等权基准不符合业界标准（Brinson 1985 原始论文用的是市值加权基准；实际应用通常用沪深 300 行业权重）
- **问题 2**: 从 `daily` 表取 sector 字段，但该字段填充可能不稳定
- **影响**: 归因结果不够精确，但作为方向性指标够用

 #### B. IC 衰减自动降级
- 5 日滚动窗口：当日 IC vs 5 日均值，衰减 >30% → active→monitoring
- 监测期 >10 天且持续衰减 → monitoring→retired
- **问题**: 5 日窗口过短——A 股日频因子 IC 波动大（中位数中波动率 ~0.05），30% 阈值在 5 日窗口下误判概率较高
- **业界做法**: 明汯用 20-60 日滚动窗口评估 IC 趋势；华泰用 12 期（月频）或 24 期（日频）滚动 IC
- **但**: 在实盘模拟场景下早期发现问题优先于精确度，较短窗口意味着更快响应，合理

 #### C. 缺失的归因维度
- **因子贡献归因**: 每个因子对当日 PnL 的贡献 ← 当前未实现，但这是业界头部机构标配
- **换手率归因**: 调仓换手 vs alpha 收益的净值影响 ← 未追踪
- **信号衰减归因**: 信号生成到执行之间的 alpha 衰减 ← 未追踪（对日频策略影响有限）

 ## 四、关键问题分析

 ### 4.1 `using` 状态的设计语义

 当前 `using = active + monitoring`，两者都参与信号生成。

 但根据 HANDOFF（ARCHITECTURE.md）及 attribution.py 的代码：
- **active** 因子因 IC 衰减会被降级到 **monitoring**（观察期 10 天）
- monitoring 期间若 IC 恢复，理应能回到 active（当前代码**未实现升回**）← 这是一个 bug/缺失
- monitoring 因子仍参与信号生成（因为 `status_filter="using"` 包含 monitoring）

 **业界做法**: 明汯/衍复等机构**明确区分** production 和 observation 因子：
- Production（对标 active）：参与交易
- Observation（对标 monitoring）：仅跟踪 IC，不参与交易

 **当前设计的问题**: monitoring 因子已被判定为 IC 衰减（质量存疑），却仍参与交易决策。这违反了"不确定时不配参与交易"的审慎原则。如果目标是尽快发现因子衰减，那 monitoring 期内应该**停止交易、仅做观察**，确认 IC 恢复后再重新启用。

 ### 4.2 样本外验证断层

 离线评估有完整的 walk-forward + CPCV + PBO（Phase 3），但在实盘模拟中：
- 盘后归因只有 5 日滚动 IC 对比
- 没有完整的 walk-forward 回测（用实盘数据持续验证因子在未知数据上的表现）

 **业界标准**: 头部量化机构（九坤、明汯）在每个交易日盘后都会做在线 walk-forward（或称 forward performance tracking / rolling OOS），用最新的交易日数据验证当前 factor set 是否继续有效。

 **这个差距不是我们做得不够好，而是我们的离线评估管线（4-6 小时）太重，不可能每天跑。需要一个轻量级在线 OOS 验证方案**。

 ### 4.3 数据量约束

 De Prado MinTRL 公式：对 A 股典型参数（SR=1.5, 偏度=-0.5, 峰度=8），MinTRL ≈ 7.8 年。
 我们当前数据量：~120 个交易日（约 6 个月）。**统计上不可靠**。

 但这源于数据可用性限制，非设计缺陷。随着数据积累，可靠性自然改善。

 ## 五、需要修改的地方

 ### 5.1 必须修改（高优先级）

 #### P1: monitoring 因子不应参与实盘交易

 **当前**: `_resolve_statuses("using")` 返回 `('active', 'monitoring')`，导致 IC 衰减后的 monitoring 因子参与信号生成。

 **建议修改**:
 1. 将 `using` 约束为仅 `('active',)` — 只有经过全评估且 IC 正常的因子参与交易
 2. monitoring 因子仅在 15:30 归因中被观察，连续 N 天 IC 恢复后自动升回 active
 3. 新增 automated 升回逻辑（当前 attribution.py 只有降级逻辑）

 **代码位置**: `factor/compute/_registry.py:16` + `quant/scheduler/attribution.py`

 #### P2: 因子升回机制缺失

 **当前**: attribution.py 有 active→monitoring 和 monitoring→retired，但**没有 monitoring→active**。

 **建议**: 在 attribution.py 中增加：
- monitoring 因子连续 5 天 IC 恢复（当前 IC 优于 5 日均值）
- → 自动升级回 active

 #### P3: 实盘在线 Walk-Forward 验证（轻量级）

 **当前**: 无在线 OOS 验证

 **建议方案**: 在 15:30 归因中新增加一个轻量级 OOS 检查：
- 不是完整的 Phase 3 CPCV（太重），而是：
- 用当天新增的 1 日数据做 expanding-window OOS IC 检查
- 对比 expanding window 内的 in-sample IC vs 当天 out-of-sample IC
- 衰减 >X% → 触发告警（不自动降级，人工审查）

 **或者更简单**: 对标明汯的做法，计算 rolling 20-60 日 OOS IC vs 满窗口 IS IC，衰减率 >50% 触发告警。

 ### 5.2 建议修改（中优先级）

 #### P4: IC 衰减窗口从 5 天扩大到 20 天

 **当前**: 5 日滚动窗口

 **问题**: A 股日频因子 IC 波动大，5 日窗口误判率较高

 **建议**: 扩大到 20 日（约 1 个月），保持 30% 阈值

 #### P5: Brinson 归因基准改为市值加权

 **当前**: 等权行业权重

 **建议**: 使用沪深 300 行业权重或全 A 市值加权

 #### P6: 盘中监控补充

- 单票/单行业集中度实时监控
- VaR 实时估算（已有 `risk/var.py`，集成到 monitor.py）
- 流动性过滤器（日均成交额 < 阈值的不交易）

 ### 5.3 可选改进（低优先级）

- DSR / MinTRL 计算（增加统计严谨性）
- 因子贡献 PnL 归因（增强盘后分析的诊断能力）
- 因子升回机制增加人工确认环节（防止频繁震荡）

 ## 六、总结

 ### 当前流程评价

 与业界标准（学术文献 + 券商金工 + 头部私募）对比，我们的实盘模拟交易流程**整体处于合理水平**，核心骨架（因子筛选 → OOS 验证 → 组合优化 → 执行 → 监控 → 归因）是完整的。

 **做得好的地方**:
 1. 信号持久化到 daily_signals 表，重启安全
 2. 零 fallback 原则（无信号拒绝执行、报价为空拒绝）
 3. 两阶段评估管线（诊断 + 正式评估）分离
 4. 自动 IC 衰减检测和降级
 5. T+1、除权、涨跌停等 A 股特殊性处理

 **需要改进的地方**:
 1. monitoring 因子参与实盘交易 → 应停止交易，仅观察
 2. 因子升回机制缺失 → monitoring→active 的路径不存在
 3. 实盘无在线 OOS 验证 → 盘后只有简单 IC 对比
 4. Brinson 归因基准不准确 → 等权 vs 市值加权
 5. 因子拥挤度检测过于简化 → 不是真正的截面相关性

 ### 最大风险

 **实盘模拟中最可能出问题的是 monitoring 因子的处理**。当前 monitoring 因子参与交易，但它们是已被判定为 IC 衰减的因子。如果 monitoring 因子较多（大量因子同时衰减），实际参与交易的 alpha 质量可能大幅下降，但系统没有机制在实盘层面预警这个情况。
