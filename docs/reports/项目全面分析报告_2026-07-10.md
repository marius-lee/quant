## 1. 项目是什么

A股量化选股系统，基于 Grinold & Kahn Fundamental Law 框架。实现因子驱动、风险中性化、组合优化、模拟执行的全流程自动化。北极星目标：¥5,000 → ¥100 万（6 个月，200 倍）。

日频运作，覆盖全 A 股 ~5,200 只，Flask 仪表盘端口 8521。

## 3. 技术栈评估与改进建议

### 当前技术合适的点

- SQLite：单用户本地零配置，千万行无压力。
- 日频：A 股 T+1 决定更高频无意义。
- 配置驱动 + "永不 fallback"：工程防御性好。

### 应补入的技术

| 缺口 | 原因 | 建议 |
|------|------|------|
| 缺乏真事件驱动回测框架 | evaluation/ 只做因子点评估，不走完整策略带约束的 walk-forward | 引入 `backtrader` 或 `zipline-reloaded` |
| 无超参自动优化 | 40+ 数值参数全靠人工调 | 引入 Optuna 做网格/贝叶斯搜索 |
| 缺市场状态检测 | 因子在牛/熊/震荡市表现截然不同，系统对所有时期一视同仁 | 引入 HMM 或滚动状态分类器，自适应因子权重 |
| 无幸存偏差修正 | sync_stock_list() 只取当前上市股票，历史退市股票缺失 | 跟踪退市股票，历史快照包含它们 |
| 单机瓶颈 | 全量 5 阶段评估耗时数小时 | 考虑 Ray 或 Dask 做分布式因子研究 |
| 无真正风险管理 | 仅有固定 ATR 参数和 5% 回撤告警，无 VaR/CVaR/压力测试 | 加入参数化 VaR、历史情景回放、相关性压力测试 |
| 无另类数据 | 仅量价+基本面 | 加入新闻情绪 NLP、社交媒体、宏观指标 |
| 无基准对比 | 不与沪深 300/中证 500 对比 | 加入每日基准 PnL 跟踪和图表 |

## 6. 架构评估——是否需要重构

**总体评分：B+（结构良好，有针对性问题）**

7 层纵向架构正确。问题出在横向：

### 需要处理的问题

**1. 双调度器问题。** 旧 `scheduler.py`（三阶段）和新 `quant/scheduler/`（5 daemon 线程）并存。旧版仍可被 import，信号生成、执行、归因逻辑均有重复。`quant/scheduler/__init__.py` 引用了 `orchestrator.py`，后者又包装旧式流程。

→ **解决**：删除 `scheduler.py`，将其唯一有用的入口（`run_loop()` CLI）迁移到 `quant/scheduler/`。

**2. `factor/compute.py` 2,998 行。** 因子函数、辅助工具、映射注册表、I/O 混在一起。

→ **拆分**：`factor/compute_price.py`、`factor/compute_fundamental.py`、保留现有的 `factor/maps.py`（_PRICE_FN_MAP / _FUNDAMENTAL_FN_MAP）。

**3. 两种配置读取方式并存。** `cfg()` 带 fallback 默认值 和 `_require_cfg()` 快速失败。约 30% 的调用点仍使用旧式 `cfg()`。

→ **统一**为 `_require_cfg`。

**4. `state_broker` 和 `state_pusher` 之间存在竞争条件。** Pipeline 通过 HTTP POST 推送状态到 Flask，Flask 从 `state_broker` direct memory 读取。两进程架构下 GET 和 POST 之间天然存在竞争。从 `web/app.py` 启动时 pipeline 和 web 本来就在同一进程内——HTTP POST 的间接层是多余的复杂性。

→ **简化**：直接内存写入。

## 7. 代码质量问题

### 高严重性

1. **`ExecutionEngine.execute()` 事务安全问题**（execution/engine.py:121）。引擎将所有订单包在一个 `BEGIN/COMMIT` 中，但循环内 `record_trade()` 会调用 `repo.get_last_buy_price()` 计算卖出 PnL。如果同一批次内先买后卖同一只股票（T+1 规则阻止了这一点，但设计脆弱），买入未提交时卖出查不到买入价。

2. **IC 衰减检测存在竞争**（quant/scheduler/attribution.py:107）。IC 衰减检查通过 raw `sqlite3.connect` 直接读取 `factor_registry`，而周度评估（`force_refresh_cache()`）可能同时在写同一张表。读写之间没有 WAL 锁协调，可能导致 `SQLITE_BUSY` 或脏读。

3. **`ocfp` 函数有重复代码**（factor/compute.py ~2700 行）。定义 `exclude_inds` 和过滤 `valid_syms` 的代码连续出现了两次——第二次赋值覆盖第一次。死代码无害，但说明删除不完整。

4. **`limit_touch_no_seal` 逐只 Python 循环**（factor/compute.py ~2600 行）。5,200 只股票用纯 Python 迭代 + 逐个 KeyError/ValueError catch。应向量化。

### 中等严重性

5. **`high52w_dist` 使用 `fundamentals["close_latest"]`**，但 `data/store.py` 的 `get_fundamentals()` 不会从 `daily` 表填充该字段。数据来自 `stocks` 表字段，在两次 `sync_stock_list` 之间可能过期。代码中注释承认了这个缺口。

6. **`_shared_limit_conn` 模块级单例**（factor/compute.py）永不显式关闭。长运行进程（web app）中，该连接可能积累过期 WAL 文件或触发 `SQLITE_BUSY`。

7. **`brinson_attribution` 缺时间下限**（quant/scheduler/attribution.py）。只按 `symbol IN (...) AND date <= ?` 过滤，没有起始日期。在积累多年历史数据的数据库上扫描行数远超必要。

### 低严重性 / 风格

8. 多处 `except Exception: pass` 吞掉真实错误（state_broker._init_state()、api_performance() 等）。应至少 `logging.warning`，或用更窄的异常类型。

9. `VERSION` 文件内容为 `"3"`，无语义化版本结构。

## 8. 算法评估——优化方向

| 算法 | 当前做法 | 问题 | 建议 |
|------|---------|------|------|
| 因子标准化 | Z-score（mean/std） | 对异常值敏感；Amihud/skewness 存在厚尾 | 对厚尾因子改用 Robust scaling（MAD），正态分布因子保留 z-score |
| 因子合成 | 等权 / IC 加权 | 忽略因子协方差；IC 加权会过度给高相关因子赋权 | 加入 IC_IR 加权 + 相关性惩罚（Qian, Hua & Sorensen 2007），或使用 Ridge 回归 |
| 协方差估计 | 自实现 Ledoit-Wolf | 工作正常但重新实现了 `sklearn.covariance.LedoitWolf` | 直接使用 sklearn 版本，减少维护成本 |
| 组合优化 | 小额资金 Top-N 等权 | 完全忽略协方差结构 | 资金 >20K 时加入最小方差 + 基数约束，用 `cvxpy` 混合整数规划 |
| 止损止盈 | 固定 ATR 倍数（2×, 3×） | 不根据波动率环境自适应 | 将 ATR 倍数设为类 VIX 市场波动率指数的函数 |
| 因子评估 IC | 120 日等权 Rank IC | 所有历史时期权重相同，忽略近期状态切换 | 对 IC 估计加入指数衰减权重（半衰期 ~60 天） |
| 基本面数据时点 | 使用最新可用季报快照 | 若未按财报公告日期筛选，会引入前视偏差 | 用 financials 表里的 `ann_date`（公告日）校验，仅使用"当时已知"的数据 |

## 总结

系统架构设计真诚扎实——7 层分离、配置纪律、"永不 fallback"原则、因子生命周期管理。对 ¥5,000→¥100 万目标的最大风险不是架构本身，而是缺少一个**端到端 walk-forward 模拟循环**，能在真金白银投入之前对整个策略做一次完整验证。加上双调度器清理和 `factor/compute.py` 拆分，是杠杆率最高的三项后续工作。


量化选股，除了基于 Grinold & Kahn Fundamental Law 框架，还有其他有效的框架吗？


• 有的，Grinold & Kahn 是"自顶向下拆解 IR"的框架，但量化选股还有其他几个有影响力的方法体系：

  1. Barra 多因子风险模型

  不直接预测收益，而是把收益方差分解到一组公共因子（行业、风格）+ 特质风险上。核心用法是风险控制而非
  alpha 生成——"我的组合在市值因子上暴露了多少？行业偏离有多大？" MSCI Barra 的 CNE5/CHE2 模型是 A 股机
  构标配。与 G&K 互补：G&K 告诉你"什么是 alpha"，Barra 告诉你"alpha 里哪些其实是 beta"。

  2. Fama-French 因子动物园

  从三因子（市场、规模、价值）演化到五因子（+盈利、投资），再到 Carhart 四因子（+动量）。核心思想是：
  所有超额收益都可以被少数系统性因子解释，alpha = 截距项。2015 年 Fama-French 中国三因子数据可直接对
  标。和 G&K 的根本区别：G&K 假设你能自建因子库，FF 假设你只能配置到已知因子。

  3. 机器学习的范式转移

  Gu, Kelly & Xiu (2020) 的实证结论重塑了这个领域：

  - GBDT/随机森林 在全 A 股截面预测上，预测 R² 是线性模型的 2-3 倍
  - 神经网络 在 >100 特征时开始显著优于树模型
  - 关键不是算法本身，而是正则化（dropout、early stopping、惩罚项）防止过拟合

  这个范式把因子工程（"我从文献里搬来 momentum、volatility、Amihud"）替换为特征工程（"我喂了 94 个公司
  特征 + 42 个宏观变量，让模型自己学交互"）。

  4. 事件驱动 / CTA 类

  不以截面对比为基础，而是做时序信号：涨停板事件、股东增减持公告、财报超预期、龙虎榜机构大买。本质
  是"某种可识别的信息事件发生后，未来 N 天有统计显著的超额收益"。这和 G&K 的因子框架正交，可以叠加。

  5. 组合优化独立体系

  - Black-Litterman：把主观观点和均衡收益结合，解决均值-方差优化对输入极端敏感的问题
  - 风险平价（Risk Parity / 桥水）：不预测收益，只平衡各类资产的风险贡献
  - Kelly 公式：信息论框架，按胜率和赔率决定仓位大小

  ———

  实际工程建议： 你现在的系统已经在 G&K 框架下跑起来了。最高收益的下一步不是换框架，而是把 Barra 风险
  分解 和 GBDT 因子合成 叠进去——前者告诉你持仓的风险来源（你的 alpha 是否只是小市值溢价），后者可能让
  你的因子组合边际 R² 跳一个台阶。然后你可以走 ML 范式的特征工程路线，从 65 个手工因子进化到自动特征交
  互。
