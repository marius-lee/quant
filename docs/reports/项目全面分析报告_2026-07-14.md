# 项目全面分析报告 — 2026-07-14

## 1. 项目是什么

A股量化选股系统，基于 Grinold & Kahn 的 Fundamental Law 框架搭建（IR = IC × √breadth）。核心理念：找到有预测力的因子 → 合成 alpha 信号 → 风险中性化 → 组合优化 → 模拟执行。

**北极星目标**: ¥5,000 → ¥100万（6个月，200倍）。所有架构决策围绕此目标。

**数据流**: 每日盘后从 tickflow/新浪/腾讯/tushare/akshare 五个数据源轮转抓取全 A 股日线 → SQLite 存储 → 70+因子计算 → Alpha 合成（sleeve/composite/intersection）→ 行业/市值中性化 → Ledoit-Wolf 协方差估计 → 资本自适应组合构建 → 模拟成交回写交易库 → 绩效归因 → Flask Web 仪表盘（端口 8521）。

## 3. 技术是否合适，需要哪些改进

### 3.1 现有技术评价

| 技术选择 | 评价 |
|----------|------|
| Python + SQLite | 适合单用户本地量化研究，够用且零运维 |
| Ledoit-Wolf 协方差 | 高维场景（5000股票×60日）最优选择，优于样本协方差 |
| Spearman Rank IC | 对异常值鲁棒，业界标准 |
| 资本自适应优化 | <2万贪心 / 2-10万得分倾斜 / >10万MV — 设计合理 |
| Flask 单进程 | 够用；无高并发需求 |

整体评价：当前技术栈的核心缺陷不在于选型，而在于**能力覆盖面的缺失**——已有框架只能回答「因子历史表现好不好」，但回答不了「实盘能不能赚钱」。

### 3.2 应优先补齐的技术

**高优先级（对实盘目标有直接贡献）**

| 缺口 | 现在状态 | 为什么重要 |
|------|----------|-----------|
| Walk-forward 样本外验证 | evaluation/ 只做因子点评估 + backtest/loop.py 做 walk-forward，但两者之间缺乏闭环 | 没有样本外验证的回测就是过拟合。ADR 029 规划了四层回测但 P1/P2 未落地 |
| 因子正交化 | 文档多次提及但未实现 | 70+因子高度相关，未正交化的因子集在合成时相当于对同一信号重复计数，alpha 会失效 |
| 幸存偏差修正 | sync_stock_list() 只取当前上市股票 | 历史数据缺少退市股，所有回测结果系统性偏高。A股每年退市 50+ 只 |
| Optuna 超参集成 | 包已引入，未深度集成 | 40+ 数值参数（窗口、阈值、权重）全靠手调。calibrate_risk_aversion() 已做网格搜索，但其他参数都硬编码 |
| HMM 市场状态检测 | 包已引入，regime/detector.py 存在但 pipeline 未使用 | 因子在不同市场状态（牛/熊/震荡）表现差异极大，当前策略在单一状态下有效就会在其他状态下失效 |

**中优先级（提升模型质量）**

| 缺口 | 说明 |
|------|------|
| Black-Litterman 融合 | 将因子得分转为投资者观点，与市场先验结合生成后验收益，优于直接 IC 加权 |
| 交易成本中的市场冲击 | 当前滑点固定千一，Almgren-Chriss 的 η 已配置但 cost.py 未使用 |
| 盘口深度 / Level-2 数据 | A 股散户众多，小额资金的重要滑点来源是流动性不足 |
| 多进程因子评估 | 70+因子串行计算耗时数小时，因子 compute 阶段仍可并行 |

### 3.3 可选量化框架补充

| 框架 | 核心思想 | 适用场景 |
|------|----------|----------|
| Grinold & Kahn Fundamental Law | IR = IC × √breadth | 因子策略 — 当前项目采用 |
| Black-Litterman | 市场均衡 + 投资者观点 → 后验收益 | 组合构建，可与因子得分融合 |
| Fama-French 多因子 | 市场/规模/价值/动量/盈利/投资 | 基准归因，风格暴露监控 |
| Barra 风险模型 | 风险因子分解 + 协方差预测 | 专业级风控，MSCI 标准 |
| Kelly Criterion | 最优下注比例 | 仓位管理 |
| 风险平价 (Risk Parity) | 等风险贡献 | 多资产配置（已部分落地） |
| HMM 状态切换 | 牛市/熊市/震荡 三状态 | 自适应因子权重 |
| 强化学习 (RL) | 状态→动作→奖励 | 端到端交易策略 |

**建议融入顺序**: Kelly（仓位管理）→ HMM（状态检测）→ Black-Litterman（组合构建）→ Barra（风险暴露监控）。

## 5. 系统架构评估

### 5.1 优点

- **分层清晰**: 7 层架构职责分明，每层有明确的对外接口和依赖方向。pipeline.py 为纯粹编排器。
- **配置驱动**: 所有参数集中于 config/config.yaml，通过 _require_cfg() 只读访问，单一真相源。
- **架构文档完整**: ARCHITECTURE.md 详细描述每层职责和接口，29 个 ADR 记录关键决策。
- **重构质量高**: TradeRepo 统一交易读写、DatabaseManager 统一连接层、scheduler 从 5 个 daemon 合并为单线程编排器——均消除了竞态条件和重复代码。

### 5.2 需要改进

**1. pipeline.py 过于庞大（530+ 行）**

`generate_signals()` 承载 Step 0-5 的所有逻辑，包含内联的 universe 过滤、止损冷却、VaR 检查等，违反单一职责原则。建议将 universe 过滤抽取到独立的 UniverseFilter，止损冷却抽取到 StopLossManager。

**2. 因子计算模块文件大小失衡**

```
fundamental.py:         1251 行  (33 个 compute_ 函数)
price/_momentum.py:      603 行
price/_event.py:         546 行
price/_alternative.py:   534 行
```

`fundamental.py` 有 33 个函数塞在一个文件，且部分函数（如 `_get_financial_historical`）是复杂的 SQL + Pandas 混合逻辑。建议按子类别拆分（质量因子、成长因子、估值因子、财务异常因子）。

**3. backtest/ 和 evaluation/ 关系不够清晰**

回测有两套引擎（backtrader + walk-forward），评估有 6 个 phase 文件。两者之间的数据流和职责边界不够明确——IC 计算在两处都可能触发，且两套引擎各有一套因子→信号流水线，增加了维护负担。

**4. data/store.py（1285 行）缺乏模块化**

`DataStore` 承担日线同步、基本面查询、涨跌停数据、基准数据、universe 管理等多种职责。应考虑按数据源拆分为多个 Repository 类（data/repos/ 已开了头）。

**5. 前端缺乏盘中实时性**

Web 依赖 SSE 推送，但 broker 只在 pipeline 各阶段结束时更新状态。盘中缺乏实时持仓变化、订单状态可见性。

## 6. 代码问题

### 6.1 逻辑问题

**1. 因子评估与实盘使用的状态不同步**

- 因子注册表管理 active / monitoring / deprecated 状态
- `generate_signals()` 使用 `status_filter="active"`（只取 active）
- 评估管线使用 `status_filter="backtesting"`（取 registered / candidate / retired）

问题：一个因子在评估中是 registered 但在实盘不在 active 集合中；反之 active 的因子可能在评估中表现不佳。两个流程之间没有自动化的晋升/降级机制，全靠脚本手动操作。

**2. pipeline.py 中的变量命名遮蔽风险**

`generate_signals()` 中段定义了 `prices = close_df.loc[risk_date].dropna()`，在 `execute_signals()` 中又定义了同名的 `prices`。当前不同代码分支，重构时容易引入 bug。

**3. IC cache 不可用时静默降级风险**

```python
# alpha/model.py
if method == "ic_weighted" and not ic_map:
    _log.info("IC cache unavailable, falling back to equal_weight")
```

当 IC cache 获取失败时静默降级到等权，意味着系统在没有信号质量信息的情况下仍会执行交易。应该在 cache 不可用时阻止交易（至少施加更严格的风控），而非静默降级。

**4. backtrader 引擎与自有引擎的代码重复**

`backtest/bt_engine.py` 的 `_SignalStrategy.next()` 内部重新调用 `generate_signals()`，等于每次 bar 都重算因子。`backtest/loop.py` 同。两套引擎各自实现了因子→信号流水线，维护成本双倍。

**5. 备份文件遗留在代码树中**

- `config/constants.py.backup-20260707-2123`
- `factor/compute.py.backup-20260707-2115`
- `factor/compute.py.bak2`
- `scripts/eval_standard.sh.backup-20260707`

建议清理移出或加入 .gitignore。

### 6.2 代码质量问题

**1. 全局可变状态**

`factor/registry.py` 有模块级共享连接 `_shared_limit_conn` + atexit 注册。`config/loader.py` 有模块级缓存。当前单线程编排器暂时安全，如果后续切换到多进程评估，SQLite 连接冲突会成为隐患。

**2. 死导入未清理**

```python
# pipeline.py line 1
import traceback  # 未使用

# execution/engine.py line 22
import traceback  # 未使用
```

**3. 测试覆盖率低**

805 行测试代码只覆盖 7 个模块（constraints、execution、factor_compute、marginal、portfolio、registry_smoke、synth），全部使用合成数据。核心模块 pipeline.py（530行）、data/store.py（1285行）、factor/stats_cache.py、alpha/model.py 均无测试覆盖。

**4. 无集成测试和端到端测试**

没有验证完整 pipeline（数据→因子→alpha→风控→优化→执行→监控）在真实数据上运行的测试。任何改动无法回归验证端到端正确性。

## 7. 算法评估

### 7.1 已有算法选择合理

| 算法 | 评价 |
|------|------|
| Spearman Rank IC | 截面因子评估的行业标准，对异常值鲁棒。正确选择。 |
| Ledoit-Wolf 收缩协方差 | 高维场景（5000股票×60日）的最优方案，优于样本协方差和因子模型。 |
| 资本自适应三级优化 | 小额资金不适合均值-方差（整手约束过于刚性），设计有工程直觉。 |
| λ 网格校准 calibrate_risk_aversion() | 替代硬编码 λ，方法正确。 |

### 7.2 可以改进的算法

**1. 因子合成方法**

当前 sleeve 模式是「每个因子独立选出 top N 股票取并集」，放弃了因子之间的互补信息。IC 加权模式将 70+ 因子压缩为单一得分向量但未考虑因子相关性。建议：
- 先做因子正交化（PCA 或 Grinold-Kahn 信息系数矩阵）
- 再用 IC_IR 加权而非 IC 加权（IC_IR 同时考虑预测力和稳定性）

**2. 协方差窗口固定**

`covariance_window: 60` 固定 60 天。在波动率急剧变化时期（如 2024 年 9 月暴涨/暴跌），60 天窗口对近期变化反应滞后。建议加入指数衰减（EWMA 协方差估计），给近期数据更高权重。

**3. 止损策略过于简单**

当前止损是 ATR 多级止损 + 固定 8% 止损混合，但未考虑：
- 止损是否基于 alpha 信号的衰减（信号消失比价格下跌更早发生）
- 波动率自适应——高波动股票 2 倍 ATR 可能只是噪音，低波动股票 2 倍 ATR 已经是趋势反转

**4. 缺少数理化的因子衰减模型**

当前因子评估输出 IC 衰减曲线，但在 pipeline 中只用当前期 IC 做权重，没有利用衰减信息预测因子的未来预测力（如用指数衰减拟合 half-life，合成时降权接近失效的因子）。

**5. 组合构建未使用 VaR 产出**

VaR 在 pipeline Step 4 只做 warning，没有反馈到组合构建中。如果 VaR 超标，应该缩小头寸或降低单票上限。
