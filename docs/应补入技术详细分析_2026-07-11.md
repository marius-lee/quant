## 1. 事件驱动回测框架 ⭐⭐⭐⭐⭐（最高优先级）

### 当前状态

evaluation/ 模块只做因子层面的点评估——IC/ICIR/half-life/PBO——不走完整策略带约束的 walk-forward。缺失的模拟维度：

- 日度组合再平衡（T+1 约束：今日买明日才能卖）
- 佣金 + 印花税 + 滑点在每笔交易上累积
- 头寸随实际资金动态调整（某天买不起所有标的时跳过部分）
- 现金拖累（未投入资金不产生收益）
- 整手约束对小微资金的刚性影响

### 方案：自定义回测循环（200 行），而非引入 backtrader/zipline

**理由**：当前 `pipeline.py` 已将信号生成（`generate_signals()`）和交易执行（`execute_signals()`）干净分离。二者接收的输入都是 `(date, capital)`，可以无侵入地塞进一个历史日期循环里。

```
for date in trading_days[start:end]:
    signals = generate_signals(date_str=date, capital=capital)
    # 这里 signals 里的 target_positions 是基于 T-1 收盘算的
    # 模拟执行放在下一个交易日，用当日 open 成交（接近真实）
    next_date = next_trading_day(date)
    result = execute_signals(signals["target_positions"], next_date)
    # 记录 equity[t] = total_wealth
```

**需要处理的关键细节**：

1. **前视偏差防护**：`generate_signals()` 里 `DataStore.get_daily(end=date)` 实际读到的是当日收盘（15:00 之后才知道），但信号应该在盘前用 T-1 收盘数据生成。修改方案：回测模式下 get_daily 的 end 参数截断到前一交易日。现有代码在 `pipeline.py` 中已做 `actual_date = data.index[-1]` 的调整，但不够严格——需要显式 offset。

2. **T+1 约束**：当前 `ExecutionEngine.execute()` 里 `check_t1()` 已做防护，但用的是"当日买入"标记。回测中需维护一个可卖出列表 = 今日持仓 - 今日新买入，直接跟踪 holding_since 字段。

3. **停牌处理**：停牌日无成交量/无 open 价。回测应跳过停牌日的卖出/买入操作，持仓暂时冻结。当前 pipeline 不做此处理——fetch 不到价格时可能误以 stale price 成交。

4. **除权除息**：`_check_ex_dividend()` 检测 ±10% 跳变并跳过买入。这只防了复权错误的数据源，但真正的除权日应该复权处理（前复权），而非跳过。

### 集成点

- 新建 `backtest/loop.py`（~200 行）
- 复用 `pipeline.generate_signals()` 和 `pipeline.execute_signals()`
- 复用 `execution/calendar.py` 的交易日历
- 复用 `data/store.py` 的 DataStore（回测模式下只读）
- 输出：equity curve CSV + 回测报告 JSON（Sharpe、MDD、年化收益、胜率、换手率）

### 工期：2-3 天

### 预期影响：**关键**。没有这个就无法验证 ¥5,000→¥100 万的路径是否在历史上成立。

## 2. 超参自动优化 ⭐⭐⭐⭐⭐

### 当前状态

config.yaml 中 40+ 个数值参数纯人工设置。无系统性搜索。示例：

```yaml
factor.evaluation.n_symbols: 500      # 为什么是 500？400 或 600 呢？
factor.evaluation.lookback: 120       # 为什么是 120？
optimizer.max_positions: 20           # 10 或 30 呢？
risk.atr_mult_stop_loss: 2.0          # 1.5 或 2.5 呢？
alpha.top_fraction: 0.3               # 0.2 或 0.4 呢？
```

### 方案：Optuna + 回测循环

Optuna 的 TPE（Tree-structured Parzen Estimator）采样器专为此类问题设计：

```
def objective(trial):
    # Optuna 建议参数
    n_symbols = trial.suggest_int('n_symbols', 200, 800, step=100)
    lookback = trial.suggest_int('lookback', 60, 250, step=30)
    max_positions = trial.suggest_int('max_positions', 5, 40, step=5)
    top_fraction = trial.suggest_float('top_fraction', 0.1, 0.5, step=0.05)
    atr_sl = trial.suggest_float('atr_sl', 1.5, 3.0, step=0.25)
    combine_mode = trial.suggest_categorical('combine_mode', ['sleeve','ic_weighted'])
    # ...

    # 写入临时 config → 跑回测 → 返回净 Sharpe
    config_override = {...}
    equity_curve = run_backtest(start='2022-01-01', end='2026-06-30',
                                 config=config_override)
    return compute_net_sharpe(equity_curve)

study = optuna.create_study(direction='maximize',
                            sampler=TPESampler(seed=42),
                            pruner=MedianPruner())
study.optimize(objective, n_trials=200)
```

**关键决策**：

1. **目标函数选什么**：净 Sharpe（扣费后）是最佳选择——同时惩罚低收益和高波动。不推荐总收益率（会奖励高杠杆/高波动参数）。可加惩罚项：若 MDD > 30% 则 Sharpe × 0.5。

2. **防止过拟合**：必须 train/test 时间分割。上例用 2022–2024 做优化、2025–2026 做验证。最终参数应该在从未见过的 2025–2026 上验证。

3. **Pruning**：`MedianPruner` 在 trial 跑到 30% 时检查——若当前 Sharpe 低于所有已完成 trial 的中位数中位数，提前终止。200 个 trial 中可能有 60% 被 prune，大幅省时间。

4. **可优化的参数范围**（建议首批 10-12 个）：
   - `factor.evaluation.n_symbols`、`lookback`
   - `alpha.combine_mode`、`top_fraction`
   - `risk.max_positions`、`max_single_position`、`covariance_window`
   - `risk.atr_mult_stop_loss`、`atr_mult_take_profit_1`
   - `optimizer.rebalance_freq`
   - `alpha.train_window`

### 集成点

- 新建 `optimizer/hyperopt.py`（~250 行）
- 依赖「事件驱动回测框架」（Gap 1），作为 objective function 的底层引擎
- 结果写回 `config/config.yaml` 的注释块（标记 `# optuna-tuned 2026-07`）

### 工期：2 天（需 Gap 1 先完成）

### 预期影响：Sharpe 可能提升 20–50%，仅靠参数搜索。

## 3. 市场状态检测 ⭐⭐⭐

### 当前状态

所有 65 个因子在牛/熊/震荡市中权重不变。但：
- **动量因子**：牛市有效、熊市反转期失效（趋势被打破时遭受 whipsaw）
- **低波动因子**：熊市超额显著、牛市跑输
- **换手率反转**：震荡市有效、趋势市平庸

### 方案：3 态 HMM + 状态条件 IC 权重

**第一步：离线训练 HMM**

用沪深 300 的 20 日收益率 + 20 日波动率训练 3 态隐马尔可夫模型（hmmlearn 库）：

```
State 0 (牛市): 正漂移 + 中低波动 — 动量/质量因子超配
State 1 (熊市): 负漂移 + 高波动   — 低波/防御/反转因子超配
State 2 (震荡): 近零漂移 + 低波动 — 反转/换手率因子超配
```

训练窗口：252 个交易日（1 年），每月重训练一次。

**第二步：在线推断**

每天收盘后用 forward filter（非 Viterbi，只用 t 时刻的观测更新信念）得到当前状态概率向量 `[P(bull), P(bear), P(sideways)]`。不用整条序列的 Viterbi 解码——那会引入前视偏差。

**第三步：自适应因子权重**

在 `alpha/model.py combine()` 中，IC 权重不再全局固定，而是按当前状态概率做加权平均：

```python
w_final[name] = sum(
    regime_prob[r] * regime_ic[name][r]
    for r in [BULL, BEAR, SIDEWAYS]
)
```

其中 `regime_ic` 是一张查找表：每个因子在每个状态下的历史平均 |IC|，存储在 `factor_regime_stats` 新表（factor_name, regime, ic_mean, n_periods），由周度评估调度器更新。

### 集成点

- 新增依赖：`hmmlearn`
- 新建 `regime/detector.py`：HMM 训练 + 在线推断（~150 行）
- 新建 SQLite 表 `factor_regime_stats`：因子 × 状态 IC 统计
- 修改 `alpha/model.py combine()`：状态条件权重（~30 行改动）
- 修改周度评估：更新 regime_stats（~40 行改动）

### 工期：2–3 天

### 预期影响：中等。因子时效性提升 10–30bp/月（业界参考）。

## 4. 幸存偏差修正 ⭐⭐⭐⭐

### 当前状态

`DataStore.sync_stock_list()` 只拉 `list_status='L'`（当前上市股票）。任何在历史日期之前已退市的股票永远不会进入数据库。回到 2020 年做因子评估时，股票池缺少 2020–2026 年间退市的 ~150 只股票，而这些退市股的因子值（大概率很差）被排除在外，导致 IC 估计虚高。

### 方案：退市股补偿 + 时点宇宙

**Part A — 数据补全（一次性）**

```
tushare pro.stock_basic(list_status='D')  →  退市股票列表
对每只退市股拉取其历史日线（上市日 → 退市日）
写入 daily 表，stocks 表标记 list_status='D' + delist_date
```

大部分退市股是 ST→\*ST→退市，数据在 tushare 可获取。一次性拉取约 600–800 只历史退市股，按 tushare 限频（200 calls/min）估算需要 30–60 分钟。

**Part B — 时点股票池（每次回测/实盘）**

在 `DataStore.get_daily()` 或新的 `get_point_in_time_universe(date)` 方法中：

```sql
SELECT symbol FROM stocks
WHERE list_date <= :date
  AND (delist_date IS NULL OR delist_date > :date)
```

当前 pipeline 的 Step 2（load symbols）已经用 `market != 'BJ'` 过滤，改成上述 SQL 即可。

**Part C — 退市日处理**

退市当天，该股票应从候选池移除。若回测持仓中持有该股，最后一个交易日以收盘价强制卖出（模拟退市清算）。

### 集成点

- `data/store.py`：新增 `sync_delisted()` 方法（~80 行）
- `data/store.py`：新增 `get_universe(date)` 方法（~20 行）
- `pipeline.py` Step 2：调用 `get_universe()` 替代当前硬编码过滤（~5 行改动）
- 回测引擎（Gap 1）：退市日强制清仓逻辑（~30 行）

### 工期：1 天

### 预期影响：**所有历史 IC 估计的真实性大幅提升**。当前所有因子评估报告都可能偏高 5–15%。

## 5. 分布式计算 ⭐⭐

### 当前状态

`factor/stats_cache.py` 中 `compute_factor_stats()` 顺序评估 65 个因子：
```
for factor in factors:
    compute_one_factor()  # ~15s each → 65 × 15s = 16min
```

P78 已将 ProcessPoolExecutor 迁移到 ThreadPoolExecutor（根除孤儿进程）。线程池用于因子值计算内部的并行，但因子之间仍是串行的。

### 方案：分批子进程（Level 1），不要上 Ray

**Level 1：multiprocessing.Pool（50 行，4–8× 加速）**

```
from multiprocessing import Pool

def evaluate_batch(factor_names):
    """子进程中独立评估一波因子。子进程有自己的 DataStore 连接。"""
    store = DataStore()
    results = {}
    for name in factor_names:
        results[name] = compute_factor_stats(store, [name], ...)
    store.close()
    return results

# 主进程
factor_names = get_factor_names()
batches = [factor_names[i::4] for i in range(4)]  # 4 个 batch
with Pool(4) as pool:
    batch_results = pool.map(evaluate_batch, batches)
```

**为什么不用 Ray**：
- Ray 的 Plasma 共享内存对 SQLite 文件无帮助——每个进程仍需自己的连接
- 当前瓶颈是 CPU（因子计算），不是内存带宽或网络 I/O
- M1 只有 4P 核，本地子进程就能榨干
- Ray 的部署和调试成本远超 4× 子进程

**关键注意事项**：
- macOS spawn 模式下，每个子进程需独立打开 SQLite WAL 连接（已验证工作）
- 如果 4 个子进程同时读 market.db，WAL 模式下读并发无锁冲突
- 但如果有写操作（如更新 factor_registry），必须等所有读进程结束后串行写入

### 集成点

- 修改 `evaluation/run_store.py`：替换 for 循环为 Pool.map（~30 行改动）
- 如果需要更细粒度，改为因子级并行（65 个 task 而非 4 个 batch）→ 负载更均衡

### 工期：半天

### 预期影响：全量评估从 ~16 分钟降到 ~4 分钟。足够日常使用，不需要复杂方案。

## 6. 真正风险管理 ⭐⭐⭐⭐

### 当前状态

| 风控手段 | 实现 | 短板 |
|----------|------|------|
| 止损 | ATR × 2 hard stop | 不区分市场状态，熊市可能每天触发 |
| 止盈 | ATR × 2 TP1 / × 3 TP2 | 强势股过早止盈 |
| 回撤告警 | 总资产 < 95% 初始 → 熔断 | 仅比较初始本金，不跟踪峰值回撤 |
| 风险度量 | 无 | 不知道组合的 VaR/CVaR/压力损失 |

### 方案：三层叠加

**Layer A：参数化 VaR + CVaR（30 行，日报）**

当前 `risk/covariance.py` 已产出 Ledoit-Wolf 协方差矩阵 Σ。在已有数据上直接算：

```python
portfolio_std = sqrt(w.T @ Sigma @ w)  # w 是权重向量
VaR_95 = portfolio_value * portfolio_std * norm.ppf(0.95)    # = 1.645
CVaR_95 = portfolio_value * portfolio_std * norm.pdf(1.645) / 0.05
```

假设正太分布是近似，但对日频组合足够实用。报告："明天有 5% 概率亏损超过 ¥XXX（VaR）；若超过，平均亏损 ¥YYY（CVaR）。"

**Layer B：历史情景回放（100 行，日报）**

定义 4 个压力情景（已有 benchmark_daily 数据）：

| 情景 | 时期 | 沪深 300 区间跌幅 | 应用方式 |
|------|------|-------------------|---------|
| 2015 股灾 | 2015-06-12 → 07-08 | -32% | 当前持仓 × 情景中各股票的实际区间收益 |
| 2018 贸易战 | 2018-01-26 → 10-18 | -29% | 同上 |
| 2020 COVID | 2020-01-20 → 02-03 | -12% | 同上 |
| 2024 流动性危机 | 2024-01-29 → 02-05 | -8% | 同上 |

实现：对每个持仓股票，取它在对应日期区间内的实际收益，加权求和得到组合情景损失。不在彼时已上市的股票用行业均值替代。

**Layer C：相关性压力测试（30 行，日报）**

模拟极端场景：所有持仓股票的相关性突变为 +1（分散化归零）：

```python
worst_loss = sum(abs(w_i) * sigma_i for each position)  # 无分散化
```

报告中显示："若所有持仓同向波动，单日最大损失 ¥XXX。"

### 集成点

- 新建 `risk/var.py`（~160 行）
- `quant/scheduler/attribution.py`：日度归因末尾调用 `risk.var.report()`（~10 行改动）
- 前端新增"风险"Tab 或叠加在 dashboard 信息栏
- 熔断条件增强：原 `总资产 < 95% 初始` 改为 `VaR_95 > 总资产 × 10% OR 情景损失 > 20%` 时才触发

### 工期：2 天

### 预期影响：不会直接提升收益，但避免灾难性损失。对于 ¥5,000→¥100 万的路径，一次 50% 回撤就宣告结束——风控是活下去的前提。

## 8. 基准对比 ⭐⭐⭐⭐⭐

from data.benchmark import get_benchmark_return  # 新函数

|------|------|------|
| 累计收益 | ∏(1+r) - 1 | 策略 vs 基准总回报 |
| Alpha | mean(strat - bench) × 252 | 年化超额收益 |
| Information Ratio | mean(alpha) / std(alpha) × √252 | 每单位跟踪误差的超额 |
| Beta | Cov(strat, bench) / Var(bench) | 市场敏感度 |
| 上行捕获率 | strat均值(r\|bench>0) / bench均值(r\|bench>0) | 牛市跟随能力 |
| 下行捕获率 | strat均值(r\|bench<0) / bench均值(r\|bench<0) | 熊市防御能力 |

**前端**：Dashboard 新增"基准"Tab。核心图表：双线累计净值（策略 vs CSI 300 vs CSI 500），时间范围可选。底部表格：上述 6 个指标。

### 集成点

- 新建 `benchmark/tracker.py`（~120 行）
- `quant/scheduler/attribution.py`：日度归因末尾调用 tracker（~20 行）
- Web：新增 `/api/benchmark` 端点 + 前端 Tab（~100 行前端）

### 工期：1–2 天

### 预期影响：零收益提升，但**立即可知策略是否在创造真正的 alpha**。大多数自建量化系统在这个检查上翻车。

## 优先级排序

按「影响力 × 可行性」排序：

| 排序 | 缺口 | 工期 | 依赖 | 为什么先做 |
|------|------|------|------|-----------|
| **1** | 基准对比（Gap 8） | 1–2 天 | 无 | 不调代码架构，不做新模型。跑完即可判断当前策略是否真 alpha。是所有后续工作的前提——如果连基准都跑不过，超参调优和状态检测没有意义。 |
| **2** | 事件驱动回测（Gap 1） | 2–3 天 | 无 | Gap 2/3 都依赖它作为 objective function。本身也是判断 ¥5,000→¥100 万可行性的唯一手段。 |
| **3** | 幸存偏差修正（Gap 4） | 1 天 | 无 | 不修这个，所有回测（来自 Gap 1/2）结果都被污染。一次性数据补全，与 Gap 1 可并行。 |
| **4** | 超参优化（Gap 2） | 2 天 | Gap 1 | 依赖回测引擎。做完后策略参数有系统性优化。 |
| **5** | 风险管理（Gap 6） | 2 天 | 无 | 独立于其他 gap。实盘安全底线。可在 Gap 1/2 并行开发。 |
| **6** | 另类数据 Phase 1（Gap 7） | 3 天 | 无 | 独立因子开发。可与 Gap 1-4 并行。 |
| **7** | 市场状态检测（Gap 3） | 2–3 天 | Gap 1 | 需用回测验证 regime-conditional 是否真的提升了 Sharpe。先跑基线回测，再加状态检测对比。 |
| **8** | 分布式计算（Gap 5） | 半天 | 无 | 纯工程优化，不影响策略质量。等因子数量增长到每次评估不够用时再做。 |

**建议节奏**：Gap 8 + Gap 4 并行（2 天）→ Gap 1（3 天）→ Gap 2（2 天）→ Gap 6 + Gap 7 并行（3 天）→ Gap 3（3 天）。总计 2 周完成全部。Gap 5 按需触发。
